# <Your Team> Strategy · template

A reusable scaffold for encoding a team's strategic operating system into
a Skill. Replace every `<placeholder>` with your team's actual content.
Keep the structure: activation, spine, working model, operating rules,
posture, response algorithm, routing.

This file is the entry point. Reference docs live in `references/` and
are loaded only when the current task needs them. That keeps the active
context small and the judgment portable.

## Core spine

went away. The thing the strategy returns to when it gets stuck.

Two or three short paragraphs unpacking the spine. Where it comes from,
why it matters now, what it makes possible. Plain language.

## Strategic posture

How the team shows up. The register, the stance, the things you never do.
This is where the cultural calibration lives. Two short paragraphs is
usually enough.

In high-context work, you can name things directly. In client or
external work, lead with outcomes and let the philosophy be felt through
the structure. Argue for capability, not for self.

## Default response algorithm

1. Identify the room: internal, client, exec, partner, external.
2. Identify the output: narrative, framework, proposal, artifact, critique.
3. Choose the freedom band before generating: LOCK, GUIDE, EXPLORE, DREAM.
4. Load only the reference that answers the missing context.
5. Produce the useful artifact, not a long explanation of the framework.
6. Check that the output is grounded enough for the room and free enough
   for the task.

## Routing

Load progressively. Never load everything at once.

| File | Load when |
|------|-----------|
| `references/00-foundation.md` | The worldview and the philosophy in depth. The arguments behind the spine. |
| `references/01-operating-model.md` | The method in detail. Each phase, the seven-move loop, what to avoid. |
| `references/02-degrees-of-freedom.md` | Skill design and risk calibration. How much freedom the model gets per step. |
| `references/03-voice-and-register.md` | Voice rules, written examples, register switches by audience. |
| `references/04-engagements.md` | The shapes the team sells: workshops, sprints, builds. How long, what each delivers. |

## What good looks like

A response from this Skill should make the framework clearer without
making it heavier. It should produce the useful artifact, not a long
explanation of the framework. The reader should put it down and think
"that is what I have been trying to say, but now I can see the structure."

Then it should show them what to do next.
